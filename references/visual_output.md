# 可视化出图模式

当用户要求「出图」「海报」「可视化攻略」「网页版」「PNG/PDF」或提供类似视觉参考图时，使用本文件。

## 输出原则

- 先完成可核验的文字攻略，再生成视觉版。视觉版不得新增未经核验的行程、价格、预约、交通或酒店结论。
- 默认生成本地 HTML 页面；如用户需要图片，再将页面截图为 PNG；如需要打印/分享，再导出 PDF。
- 图片素材优先使用用户提供图片、真实景点/酒店/交通图或可访问的公开图片。若使用 AI 生成图片，必须在图注或来源区标注「AI 生成氛围图」或「AI 生成路线插图」。
- 路线插图只能表达氛围和相对结构，不替代高德/百度/官方地图导航；涉及实际导航时必须提醒以地图 App 为准。
- 所有视觉页保留来源区和「需最终确认」标注，不因为做成图片就省略风险提示。

## 交付物

建议目录：

```text
travel-plan-output/
├── index.html
├── assets/
│   ├── hero.png 或 hero.jpg
│   └── route-illustration.png
├── screenshots/
│   ├── desktop.png
│   └── mobile.png
└── travel-plan.pdf（可选）
```

最终回复给用户时提供本地文件链接：`index.html`、主截图 PNG、PDF（如生成）。

## 视觉结构

参考网页型旅行海报，而不是普通 Markdown 表格截图。

页面建议包含：

1. **Hero 首屏**
   - 全宽大图背景
   - 目的地和天数作为最大标题
   - 旅行主题副标题，例如「吃喝打卡」「海岛周末」「户外自驾」
   - 标签：天数、交通方式、人数/房间、住宿区域、预算

2. **顶部导航**
   - 总览
   - 每日行程
   - 住宿与交通
   - 吃喝打卡
   - 预约/确认清单
   - 来源

3. **路线总览**
   - 左侧放真实图片或 AI 路线氛围图
   - 右侧用信息卡总结住宿锚点、硬需求、不纳入项、强度、风险

4. **每日行程**
   - Day 1 / Day 2 / Day 3 分段
   - 时间轴展示交通、吃饭、打卡、休息
   - 每天附小红书搜索入口和雨天备选

5. **住宿与交通**
   - 酒店只放一个主来源链接和推荐理由
   - 交通必须对比高铁/飞机/自驾，说明推荐原因
   - 票价、余票、房价、余房保留「需最终确认」

6. **吃喝打卡**
   - 按区域组织
   - 附小红书搜索结果页入口，不默认抓取单帖

7. **预约/确认清单**
   - 车票
   - 酒店
   - 景点预约
   - 天气复查
   - 行李和证件

8. **来源**
   - 官方/平台链接
   - 小红书搜索入口
   - 交通和酒店主来源
   - 查询日期

## 设计规范

- 风格根据目的地调整：海岛可清爽明亮，城市吃喝可更紧凑，户外自驾可使用大图和路线感。
- 不要把正文塞进图片导致难读；用卡片、分栏、时间轴拆开。
- 文字必须在桌面和手机宽度都不溢出、不重叠。
- 中文字体使用系统 sans-serif，例如 `-apple-system, BlinkMacSystemFont, "Segoe UI", "Microsoft YaHei", sans-serif`。
- 避免纯装饰渐变和空洞大卡片；视觉资产要和目的地/路线相关。
- 大图上叠文字时必须加暗色遮罩，保证标题可读。
- 不要把 AI 生成图伪装成真实路线或真实景点照片。

## 技术流程

1. 创建输出目录。
2. 准备图片素材：
   - 用户有参考图或实拍图时优先使用。
   - 没有合适图片时，可生成 hero/route 氛围图，并在页面标注 AI 生成。
3. 编写 `index.html`，内联或配套 CSS。
4. 用浏览器/截图工具检查桌面和手机尺寸：
   - 1440px 宽桌面
   - 390px 宽手机
5. 修复文字溢出、遮挡、图片比例错误。
6. 导出 PNG/PDF。

## HTML 数据来源

视觉版的数据必须来自前面已经确认的攻略结构：

- 目的地、日期、人数、房间、预算
- 交通推荐及排除理由
- 住宿锚点和主来源链接
- 点位和小红书搜索入口
- 每日行程和缓冲时间
- 预约/复核清单
- 来源和可信度

如果某项数据缺失，用「待确认」显示，不要为了画面完整而编造。
